# Agent Eval — 项目故事

## Inspiration

2026 年初，大模型 Agent 正在从「能用」走向「好用」，但一个根本问题始终悬而未决：**你怎么知道你的 Agent 变好了还是变差了？**

改了一行 prompt、换了一个模型、调整了工具调用策略——这些变更对端到端质量的影响，在大多数团队里靠的是「感觉」和「跑几个 case 看看」。没有量化的评测体系，每一次 Agent 迭代都像在盲飞。

我们见过太多这样的场景：一个看似无害的 prompt 微调，让意图识别准确率暴跌 15%；一次模型升级，幻觉率不降反升；新增一个召回源，结果把相关度最高的文档挤出了 Top 5。更糟糕的是，这些问题往往要到用户投诉才知道。

**评测不该是「上线前跑一下」的一次性动作，而应该是 Agent 的全生命周期伴侣。** 这就是 Agent Eval 项目的起点——构建一套面向大模型 Agent 调用链路的自动化评测平台，让每一层的行为都可度量、每一次变更的影响都可追溯、每一个版本的退化都可预警。

---

## What it does

Agent Eval 是一套**全链路、分层式、持续化**的 Agent 评测系统。它不只是一个「打分器」，而是覆盖了从数据采集到版本对比的完整闭环：

### 五层分层评测

对 Agent 执行链路的每一层进行独立度量：

| 评测层 | 评测对象 | 核心指标 |
|--------|---------|---------|
| **意图层** | 用户意图分类是否准确 | 意图匹配精度、命名实体识别 F1、置信度校准 |
| **召回层** | 检索结果的精度与覆盖 | Precision@K、Recall@K、MRR、NDCG、多样性 |
| **工具层** | 工具选择与参数正确性 | 工具选择精度、参数准确性、调用序列正确性、成功率 |
| **生成层** | 答案的事实性与完整性 | 事实准确性、完整性、幻觉检测、语义相似度、语言质量 |
| **效果层** | 端到端任务完成度 | 任务完成度、延迟评分、Token 效率、错误恢复能力 |

### 双模数据采集

- **自研 SDK**：4 行代码嵌入 Agent 管道，精确控制 Span 粒度，支持 `intent` → `retrieval` → `tool_call` → `generation` 四阶段上报
- **OpenTelemetry 适配器**：零侵入接入 LangChain / LlamaIndex / OpenAI Agents SDK 等主流框架，通过三层映射策略自动推导 Span 类型

### 智能评测集积累

通过「LLM 定时抽样 → 置信度分流 → 人工兜底」的复合管线，将生产环境 Trace 自动转化为长期评测用例，评测集越用越丰富。

### 科学版本对比

基于 Paired t-test 的配对样本检验 + Cohen's d 效应量 + Bootstrap 置信区间，不只看均值变化，还能判断差异是否为统计显著、退化程度有多大。

### 7×24 Agent Runtime

系统不仅评测 Agent，自己也是一个持续在线的 Agent 服务——通过 Telegram Gateway 接收消息，Brain 模块做意图解析与命令执行，Scheduler 管理定时抽样和日报生成。

## How we built it

### 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 后端框架 | Python + FastAPI | 异步原生支持，与 asyncio 生态无缝集成 |
| 数据库 | PostgreSQL 16 + JSONB | 结构化字段走 B-tree 索引，Span 的动态结构用 JSONB 存储 |
| 消息队列 | Redis List → Kafka（规划中） | V1 轻量启动，V2 支持多消费者 |
| 异步任务 | Celery + asyncio | 评测异步化，API 不阻塞 |
| 评测 LLM | Qwen / GPT-4o | 固定 temperature=0 确保结果可复现 |
| 前端 | 内嵌 Dashboard（HTML + vanilla JS） | 极简部署，无需额外前端项目 |
| 部署 | Docker Compose | 一键启动 PostgreSQL + Redis |

### 架构设计

系统采用**五层架构**：

```
接入层（FastAPI + Telegram Gateway）
    ↓
调度层（Celery 任务队列 + Scheduler 定时器）
    ↓
执行层（Agent 调用 → SDK 上报 → Ingest 消费 → 五层评测器）
    ↓
存储层（PostgreSQL + JSONB，traces/spans/scores 完整持久化）
    ↓
分析层（聚合 → 版本对比 → 退化告警 → 报告导出）
```

核心设计决策包括：

1. **分阶段事件流而非一次性上报**：Agent 边执行边上报告，评测系统实时组装，Agent 无需缓存全链路数据
2. **评测器插件架构**：五层评测器通过注册中心管理，支持多版本共存、动态切换、独立升级
3. **LLM-as-Judge 的多次采样 + 降级兜底**：关键维度采样 3 次取均值，Token 超预算自动切 mini 模型
4. **挂载模式**：评测系统可作为 Python 对象内嵌在 Agent 进程中，开发调试零额外运维

## Challenges we ran into

### 1. LLM-as-Judge 的一致性困境

用 LLM 评测 LLM，最大的问题是**评分不可复现**。同一个回答，GPT-4o 今天打 85 分，明天可能打 78 分。我们通过三重策略应对：

- **temperature=0 固定** + 多次采样取均值
- **双重验证**：关键维度（幻觉检测）同时用确定性和 LLM 两种方式交叉校验
- **评测器版本锁定**：每次评测记录 `evaluator_version`，版本对比强制要求评测器版本一致

即便如此，LLM Judge 的方差仍然是一个持续优化的方向——我们在设计中预留了定期人工校准和 Cohen's Kappa 一致性检验的机制。

### 2. 无期望值时的评测退化

生产环境的 Trace 没有预先标注的期望值。召回层的 Precision@K、意图层的 NERAccuracy 等指标完全依赖 `expected_*` 数据，缺失时只能跳过。

我们设计了一套「**感知 → 跳过 → 重归一化**」的自适应公式系统：根据期望值的实际可用性动态决定哪些维度参与计算，剩余维度权重按比例重新分配。效果层几乎全维度在无期望值模式下仍可用，生成层大部分维度可通过 LLM 自身知识降级判断。

### 3. OTel Span 到评测层的映射

不同 Agent 框架产生的 Span 命名千差万别——LangChain 叫 `ChatOpenAI`，LlamaIndex 叫 `llm_predict`，自研框架可能叫任何东西。将它们统一映射到 `intent/retrieval/tool_call/generation` 四个标准类型，需要兼顾精确性和兼容性。

最终方案是**三层策略**：属性优先（`eval.span_type` 显式标注）→ 模式匹配（YAML 映射表 + 最长匹配原则）→ 兜底（返回原始 span.name）。配合 YAML + 硬编码双重保障，即使配置文件损坏也不影响运行。

### 4. 评测异步化的竞态问题

将同步评测改为异步后，出现了经典的「取错 Run」竞态：同一 Trace 被连续提交多次评测时，取「最新一条 EvalRun」可能拿到并发中尚未完成的另一条。

解决方案很直接但容易被忽略：**通过 `eval_run_id` 精确定位**，而不是依赖时间排序。同时 EvalScore 通过 `eval_run_id` 外键关联，支持同一 Trace 的多轮评分独立记录、完整追溯。

### 5. 评测器自身的版本管理

评测器升级会导致同一 Trace 得分变化——如果昨天用 v1.0 评测，今天用 v1.1 评测，分数不可比。这直接破坏了版本对比的公平性。

我们为评测器引入了完整的 SemVer 版本管理体系：每次评测记录 `evaluator_version`，版本对比时强制一致性检查。Bug 修复 bump PATCH，新维度 bump MINOR，框架重构 bump MAJOR。旧版和新版可并行运行，提供「重评」功能让历史数据用新版评测器重新打分。

## Accomplishments that we're proud of

### 完整的数据闭环

从 Agent 执行 → SDK 上报 → Ingest 消费 → 五层评测 → 得分入库 → 版本对比，每一步都有数据落表、每一步都可追溯。评测系统自身产生的数据（eval_scores）本身也是结构化、可分析的。

### 自适应公式系统

不是死板套用固定公式，而是根据期望值的实际内容动态调整：`mode: "any"` 时退化为二值判断，`divergent_ok: true` 时跳过完整性检查，`nice_to_have` 检查点只加分不扣分。这让同一套评测器能处理从「严格的事实问答」到「发散的推荐建议」等各类场景。

### LLM 自动化评测集积累管线

将「被动等标注」升级为「主动采样 + 置信度分流 + 人工兜底」的产线化流程：每日定时分层抽样 → LLM 批量标注并输出置信度 → `≥ 0.9` 自动入库，`0.6~0.9` 入审核队列，`< 0.6` 暂存候选池。评测集从此不再是瓶颈。

### Agent Runtime：从「评测工具」到「可评测的服务」

系统不仅能评测别人，自己也跑着一个 7×24 的 Telegram Agent——通过日常对话积累生产 Trace，这些 Trace 又通过采样管线转化为评测集，形成「**自己生产数据 → 自己评测自己 → 自己迭代自己**」的飞轮。

### 零侵入 OTel 适配

不需要 Agent 框架做任何代码改动，只需注册一个 Exporter，LangChain / LlamaIndex 的自动埋点 Span 就自动流入评测系统。三层映射策略覆盖了主流框架的核心 Span 类型，对未知框架还能通过自定义映射表快速接入。

### 超过 2800 行测试代码

尤其在 Agent Runtime 模块——7×24 常驻服务不像 HTTP 接口有超时兜底，测试必须更充分。5 个测试文件覆盖了意图解析、消息路由、Brain 执行器、Scheduler 调度、Runtime 生命周期，测试行数超过了第一次提交的整个项目规模。

## What we learned

### 评测系统的「元问题」

评测系统本身也需要被评测。LLM Judge 的评分是否可靠？评测集的标注偏差有多大？同一 Case 在不同评测器版本下的得分是否一致？这些问题我们花了大量精力在设计阶段就想清楚——标注来源追踪（`annotation_method` JSONB）、评测器版本锁定、定期人工校准、Case 失效检测，这些都是评测系统不能回避的元问题。

### 先文档后代码是正确的

项目遵循严格的 7 阶段开发工作流——需求分析 → 方案设计 → 门禁审查 → 开发 → 代码审查 → 测试验证 → 交付总结。实践下来，最「浪费时间」的文档阶段反而是最高效的投资：一份好的方案设计能在编码前消灭 80% 的返工。

### 分层优于大统一

最开始想做一个「万能评测器」——输入 query + response，输出一个分数。但 Agent 不是黑盒，它的行为分布在意图、召回、工具、生成四个阶段，每个阶段的「好」有完全不同的定义。五层分层评测看似复杂，实则让每一层的评测逻辑都变得清晰可维护。

### JSONB 是 Agent 数据的天然选择

Agent 链路的内部结构随版本迭代频繁变化——新增工具类型、变更意图分类体系、调整召回策略。纯关系型 schema 的迁移成本太高，JSONB + schema-on-read 的策略让数据模型保持弹性，配合 GIN 索引仍然可以高效查询。

### Redis 作为缓冲层是明智的

SDK 不直接写 PostgreSQL，而是通过 Redis List 做缓冲，Ingest 消费者独立拉取批量写入。这个设计让 Agent 侧的上报几乎零延迟（O(1) RPUSH），同时避免数据库连接池被大量并发上报打满。进程崩溃也不丢数据——Redis AOF 兜底。

## What's next for Agent Eval

### 生产级消息队列升级

当前 Redis List 适合 V1 的轻量场景。随着采样量和并发 Agent 数增长，计划迁移到 Kafka——支持消费者组、消息持久化、水平扩展。

### 评测 LLM 的自我校准

引入定期人工标注校准 + Cohen's Kappa 一致性检验的自动化流水线。目标：当 LLM Judge 与人工评分的偏差超过阈值时，自动触发 Prompt 调优或模型切换。

### Dashboard 可视化增强

从当前的 HTML 内嵌看板升级为 React 前端：总分雷达图、各层得分箱线图、版本趋势折线图、退化 Case 热力图。让评测结果不仅是数据，更是可交互的洞察。

### 多模态 Agent 评测

当前系统聚焦文本链路。随着多模态 Agent（图像理解 + 文本生成）的普及，需要扩展 Span 协议支持图像输入、支持视觉问答的事实准确性评测。

### 社区化与开放

SDK 和评测器协议已有清晰的边界。计划将 SDK 和 OTel 适配器作为独立开源包发布，让任何 Agent 项目都能以最低成本接入评测体系。评测器的 Prompt 模板也计划开放社区贡献。

### 成本感知评测

将 Token 消耗、API 调用费用纳入评测维度，在「质量—成本」的帕累托前沿上做决策。不只回答「哪个版本更好」，还回答「多花的钱值不值」。
